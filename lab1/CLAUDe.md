# Software Requirements Specification (SRS)

## 1. บทนำ

### 1.1 วัตถุประสงค์
เอกสารนี้สรุปข้อกำหนดของระบบเว็บสำหรับ "ธนาคารขยะ" ซึ่งเป็นระบบแจ้งเตือนและจัดการการปฏิบัติงาน โดยอ้างอิงจากข้อมูลใน `readme.md` ภายในโปรเจค

### 1.2 ขอบเขตของระบบ
ระบบจะให้บริการเว็บแอปที่รองรับการใช้งานทั้งบนมือถือและเครื่องคอมพิวเตอร์ (PC) สำหรับผู้ใช้งานสองระดับคือ ผู้จัดการและนักปฏิบัติการ และใช้ฐานข้อมูล SQLite เพื่อเก็บข้อมูล

## 2. คำอธิบายภาพรวม

### 2.1 มุมมองของผลิตภัณฑ์
ระบบเป็นเว็บแอปแบบไคลเอนต์-เซิร์ฟเวอร์ ที่มีฟังก์ชันหลัก: ลงชื่อเข้าใช้, แจ้งเตือน, และแดชบอร์ดการจัดการ/ปฏิบัติงาน

### 2.2 ผู้ใช้งานและบทบาท
- ผู้จัดการ: เข้าถึงแดชบอร์ดสรุปข้อมูล, จัดการผู้ใช้, ดูสถิติ และส่ง/กำหนดการแจ้งเตือน
- นักปฏิบัติการ: รับแจ้งเตือน, อัพเดตสถานะงาน, ดูงานที่ได้รับมอบหมาย

### 2.3 ข้อจำกัด
- ฐานข้อมูล: ใช้ SQLite
- รองรับอุปกรณ์: มือถือและ PC (responsive design)

## 3. ข้อกำหนดเชิงฟังก์ชัน (Functional Requirements)

### FR-1: ระบบยืนยันตัวตน
- ผู้ใช้งานต้องสามารถลงชื่อเข้าใช้ด้วยบัญชีผู้ใช้และรหัสผ่าน
- แยกระดับสิทธิ์ระหว่าง `Manager` และ `Operator`

### FR-2: แจ้งเตือน
- ระบบต้องสามารถส่ง/แสดงการแจ้งเตือนไปยังนักปฏิบัติการ (แบบในแอป)
- ผู้จัดการสามารถสร้าง/ลบ/แก้ไขการแจ้งเตือน

### FR-3: การจัดการงาน
- นักปฏิบัติการสามารถรับมอบหมายงานและอัปเดตสถานะ (เช่น: รอดำเนินการ, กำลังดำเนินการ, เสร็จสิ้น)
- ผู้จัดการสามารถมอบหมายงานและดูสถานะงานทั้งหมด

### FR-4: อินเทอร์เฟซที่ตอบสนอง
- UI ต้องรองรับการแสดงผลทั้งบนอุปกรณ์มือถือและเดสก์ท็อป (responsive)

### FR-5: การจัดเก็บข้อมูล
- ข้อมูลผู้ใช้ ข้อความแจ้งเตือน และประวัติการทำงาน ต้องถูกเก็บในฐานข้อมูล SQLite

## 4. ข้อกำหนดไม่เชิงฟังก์ชัน (Non-Functional Requirements)

- NFR-1 (ความพร้อมใช้งาน): ระบบต้องพร้อมใช้งานสำหรับผู้ใช้ทั่วไปในช่วงเวลาทำการและรองรับการเชื่อมต่อพร้อมกันจำนวนเล็ก ๆ (ขึ้นกับ SQLite)
- NFR-2 (ประสิทธิภาพ): หน้าเว็บหลักต้องโหลดภายใน 2–3 วินาทีบนการเชื่อมต่อปกติ
- NFR-3 (ความปลอดภัย): รหัสผ่านต้องถูกเก็บแบบแฮช; มีการตรวจสอบสิทธิ์ก่อนเข้าถึง API ที่สำคัญ
- NFR-4 (ความเข้ากันได้): รองรับเบราว์เซอร์ยอดนิยมและ viewport ของมือถือทั่วไป
- NFR-5 (ความสามารถในการบำรุงรักษา): โค้ดต้องแยกชั้น UI / โลจิก / ดาต้าให้ชัดเจน

## 5. กรณีการใช้งานหลัก (Use Cases)

- UC-1: ผู้ใช้ลงชื่อเข้าใช้ -> ระบบตรวจสอบสิทธิ์ -> แสดงแดชบอร์ดตามบทบาท
- UC-2: ผู้จัดการสร้างแจ้งเตือน -> นักปฏิบัติการได้รับแจ้งในรายการแจ้งเตือน
- UC-3: นักปฏิบัติการอัปเดตสถานะงาน -> สถานะปรากฏบนแดชบอร์ดผู้จัดการ

## 6. ข้อกำหนดข้อมูล (Data Requirements)

- ตารางผู้ใช้: id, username, password_hash, role, metadata
- ตารางการแจ้งเตือน: id, title, message, created_by, created_at, target
- ตารางงาน: id, title, description, assigned_to, status, timestamps
- ฐานข้อมูลทั้งหมดจัดเก็บในไฟล์ SQLite ในนามที่กำหนดใน environment/config

## 7. ข้อสมมติและพึ่งพา

- สมมติว่าผู้ใช้มีการเชื่อมต่ออินเทอร์เน็ตขณะใช้งาน
- พึ่งพา SQLite เป็น storage ชั้นหลัก (ไม่ใช้ DBMS ที่รองรับ concurrency ระดับสูง)

## 8. ความเสี่ยง

- SQLite อาจจำกัดความสามารถในการรองรับ concurrent writes จำนวนมาก
- ต้องป้องกันข้อมูลสำคัญ (รหัสผ่าน, โทเคน) ให้เหมาะสม

## 9. ข้อกำหนดเพิ่มเติม / ข้อเสนอแนะ

- หากต้องการรองรับผู้ใช้จำนวนมาก ควรพิจารณาย้ายเป็น PostgreSQL/MySQL
- พิจารณาเพิ่มกลไกการส่งแจ้งเตือนแบบ push (เช่น FCM) ในอนาคต

## 10. ภาคผนวก / คำศัพท์

- Manager: ผู้จัดการระบบ/ผู้บังคับบัญชา
- Operator: ผู้ปฏิบัติงานภาคสนาม
